  3. cowtrix

    Hey ZTIF - simply put, it's the beginnings of these things. Mineable nodes, Ais, vehicles, are all networked objects.

    A simple use case - say I wanted a mod that spawned purple cubes all around the map. This would be pretty much impossible right now. You can make a bunch of cubes on the server through scripting, sure, but the client has no way to know about them. It'll just run into invisible colliders. This is because both the server and the client need to know and agree what things are, where they are, and what state they are in. Now, with these changes, you can make a mod for the Steam Workshop with a purple cube in it, add that mod to a server, and you'll be able to spawn purple cubes all over the place. All clients will be able to see them and agree what/where they are, collision will work properly, and nice things like that.

    Now, we're slowly rolling out more functionality with these networked objects, but we can't rush it. When we push something out into the SDK, we're locking ourselves into a structure for that thing. We need to start thinking about things like backwards compatibility, and that's a whole ordeal. We're happy to do that with Construction stuff right now, because that's a system that's close to maturity. But things like vehicles, machines, minerals, all stuff which might be fundamentally changing in the near future, well there's not much point putting that into the SDK yet.

    The first functionality, which should be good to go for this next patch and is probably the most general one, is what you refer to. Networked rigidbodies! This is a type of networked object, not all networked objects that you make have to be rigidbodies. But Networked Rigidbodies will do stuff like sync their position between server and client, and simulate physics if you want them to. You can make a big stack of purple cubes in your server and have them fall over!
