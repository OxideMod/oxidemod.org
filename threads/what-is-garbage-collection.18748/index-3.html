  3. sqroot

    A garbage collector looks for unused memory on the heap (dynamic memory) and frees it (tells the OS that this memory may be used again).
    There are several different techniques for garbage collection.
    The one used in Unity is a "stop the world" garbage collector.
    Put simply, it tracks all references to dynamic memory and marks these memory areas as "used". Next it traverses the dynamic memory and frees every memory area that isn't marked as "used".
    Applications may not write to memory while this is happening, because then memory might get added but not marked as "used", resulting in used memory being freed.
    To achieve this, the garbage collector "stops the world", meaning the entire application halts until all unused memory has been freed.
    This is what you're seeing in your console, and because Rust:IO uses OnTick, the chance that a garbage collection occurs while Rust:IO is running is relatively high, as opposed to plugins that are only run once every few thousand ticks because they were called by a user command or similar.

    Garbage collectors are used so that programmers do not manually have to manage dynamic memory. If they forget to free dynamic memory in some case, a memory leak occurs: The amount of memory needed by the application increases until eventually all memory is used up.

    In reality the entire thing is a bit more complicated. GCs use relatively complicated algorithms to reduce the pause time (guard routines, colors, generations, ...) as well or use an algorithm that doesn't pause at all, at the cost of more memory leaks being possible (reference counting for example).

    tl;dr: Nothing you have to worry about (unless you're using a VPS with distributed memory - this might greatly slow down garbage collection).
